## 1. 設計思想

### 1.1 基本原則

- **FujiTraceの着火剤である。** チャットbot自体で利益を取らない。全ての設計判断はFujiTrace普及に寄与するかで評価する
- **量産可能な構造にする。** 顧客ごとのカスタマイズは最小限（ドキュメント・デザイン・トーン設定のみ）
- **FujiTraceは空気のように存在する。** 顧客がFujiTraceの設定や導入作業を意識することは一切ない

### 1.2 FujiTrace統合の核心: 「組み込みは1回だけ」

現行のFujiTrace（プロキシ型）と着火剤チャットbotでは、FujiTraceの統合方式が根本的に異なる:

```
現行モデル（プロキシ型）:
  顧客のコード → FujiTraceプロキシURL → LLM API
  （顧客が毎回URLを設定する必要がある）

着火剤モデル（自前チャットbot）:
  チャットbotエンジン（共通） → LLM API
       ↑
  FujiTraceはエンジン内部に1回だけ組み込み済み
       ↑
  顧客ごとに変わるのはドキュメント・デザインのみ
```

FujiTraceの監視コードはチャットbotエンジンのコアに**1回だけ**実装する。新しい顧客が追加されるたびに、そのチャットbotは**自動的に**FujiTrace監視下に入る。顧客側の作業はゼロ。

### 2.2 顧客ごとに変わる部分 vs 共通部分

| 顧客ごとに変わる | 全顧客で共通 |
|---------------|-----------|
| RAGドキュメント（FAQ、商品情報等） | チャットbotエンジン |
| ウィジェットのデザイン（ロゴ・カラー） | LLM API呼び出しロジック |
| 回答トーン設定（丁寧語・カジュアル等） | RAGパイプライン（チャンク分割・検索） |
| システムプロンプト（業種別テンプレート） | FujiTrace監視（トレース・品質・コスト） |
| ドメイン/サブドメイン設定 | ウィジェット埋め込みスクリプト |

### 2.3 完全自動化: 申し込みから本番稼働まで人手ゼロ

> **設計上の鉄則: 人手が介在するポイントをゼロにする。**
> 「API実費のみ」で提供する以上、1社あたりに人手をかけたら赤字になる。
> 導入の完全自動化は「あったら嬉しい」ではなく、**ビジネスモデルの存続条件**。

#### 完全自動化フロー

#### なぜ完全自動化が必須か

| | 競合（KARAKURI等） | FujiTrace着火剤 |
|--|------------------|----------------|
| 導入プロセス | 営業→要件定義→設定→テスト（数週間） | セルフサービスで即日（数分） |
| 1社あたりの人件費 | 営業・CS・エンジニアが必要 | ゼロ |
| スケーラビリティ | 人手に比例（線形） | サーバーに比例（無限） |
| 限界費用 | 高い（人を増やさないとスケールしない） | ほぼゼロ（何社増えてもコスト変わらず） |

**人手ゼロ = 限界費用ゼロ = 何社増えてもコストが変わらない。**
だから「API実費のみ」という料金モデルが持続可能になる。

これはStripeが「開発者がコードを貼るだけで決済が動く」を実現したのと同じ構造。
導入に人手がかかる競合は、価格を下げても人件費が消えないため、この構造に追随できない。

#### 自動化の副次的効果

1. **広告との相性が最高** — 広告「30秒で試せます」→ 実際に数分で本番稼働 → トライアルから本番への壁がゼロ
2. **量産が無限** — 100社でも10,000社でも、サーバーリソースが許す限りスケール
3. **深夜・休日でも導入可能** — 顧客が思い立った瞬間に始められる（営業時間の制約なし）
4. **データ蓄積が即時開始** — 導入した瞬間からFujiTraceにトレースデータが流れ始める

## 3. MVPスコープ

### 3.1 MVP機能（Phase 0）

3機能に絞る。これ以外はPhase 1以降。

#### 機能① Webチャットウィジェット

- HPに `<script>` タグ1行で埋め込み可能
- レスポンシブデザイン（PC・スマホ対応）
- カスタマイズ: ロゴ、アクセントカラー、表示位置（右下/左下）、ウェルカムメッセージ
- チャット履歴の保持（セッション内）

#### 機能② RAG（検索拡張生成）

- ドキュメントアップロード: PDF、テキスト、FAQ（CSV/JSON）
- チャンク分割 → ベクトル化 → ベクトルDB格納
- 質問に対して関連チャンクを検索 → LLMに渡して回答生成
- 回答にソース（参照元ドキュメント）を表示

#### 機能③ FujiTrace監視ダッシュボード統合

- チャットbot管理画面は**FujiTraceダッシュボードに統合**（専用管理画面は作らない）
- FujiTraceダッシュボードから:
  - チャットbotの設定変更（ドキュメント追加・デザイン変更）
  - 回答品質スコアの確認
  - トークンコスト（円換算）の確認
  - 全会話トレースの閲覧
  - ハルシネーション検知アラート

## 4. 技術スタック（検討中）

> このセクションはMVP開発開始時に確定する。現時点では方針のみ記載。

### 4.1 方針

- **自前開発**（OSSプラットフォーム依存を排除）
- FujiTrace本体と同じ技術スタックを極力使い、学習コスト・メンテナンスコストを最小化
- マルチテナント対応を最初から設計に組み込む

### 4.2 技術スタック候補

| レイヤー | 候補 | 備考 |
|---------|------|------|
| バックエンド | Fastify 5 + TypeScript（ESM） | FujiTrace本体と同一 |
| フロントエンド（管理画面） | FujiTraceダッシュボードに統合 | 新規UI不要 |
| チャットウィジェット | 軽量Web Component or iframe | 顧客HPへの影響を最小化 |
| RAG - ベクトルDB | 未定（pgvector / Qdrant / Chroma等） | コスト・運用性で選定 |
| RAG - Embedding | 未定（OpenAI / Cohere / 国産等） | コスト・日本語精度で選定 |
| LLM | マルチプロバイダー対応（OpenAI / Anthropic / 国産） | 顧客が選択可能にする |
| DB | PostgreSQL（FujiTraceと共有 or 分離） | マルチテナントのデータ分離方式を検討 |
| ストレージ | S3互換（ドキュメント保存用） | R2 / MinIO等 |

### 4.3 FujiTrace統合の技術的ポイント

FujiTraceのプロキシ型アーキテクチャをチャットbotエンジン内部で利用する:

```
チャットbotエンジン内部:
  ユーザーの質問
    → RAG検索（関連ドキュメント取得）
    → プロンプト構築
    → FujiTraceプロキシ経由でLLM API呼び出し  ← ここで自動的にトレース記録
    → レスポンスをユーザーに返却
    → FujiTraceが品質スコア・コスト・レイテンシを自動記録
```

既存のFujiTraceプロキシハンドラー（`src/proxy/handler.ts`）をそのまま内部呼び出しすることで、新規の監視コード開発が不要になる。

### 4.4 内蔵APIキーモデル: 顧客にAPIキー発行させない（2026-03-29追加）

> **海外競合はBYOK（Bring Your Own Key）モデル = 顧客がOpenAI等のAPIキーを自分で発行して持ち込む。FujiTraceはこれをやらない。**

#### 海外競合との構造比較

```
海外競合のBYOKモデル:
  顧客がOpenAIアカウント作成
    → APIキー発行
      → 海外企業のチャットbotツールにキーを貼り付け
        → ツール経由でLLM呼び出し

  問題1: 顧客のAPIキーが第三者ツールに渡る（セキュリティリスク）
  問題2: 顧客にAPIキー管理の知識が必要（導入ハードル）
  問題3: OpenAIアカウント作成が脱落ポイント（特にITリテラシー低い層）


FujiTrace内蔵キーモデル:
  顧客はチャットbotを作るだけ
    → FujiTraceバックエンド（サーバーサイド）がLLM APIを呼ぶ
      → APIキーはサーバー内部に保持、顧客には一切見えない

  顧客が触れるもの: ダッシュボードのUI のみ
  顧客が触れないもの: APIキー、LLM API直接通信
```

**APIキーはFujiTraceのサーバー内部に1つだけ存在し、顧客側には一切露出しない。**
Stripeの加盟店が決済ネットワークの認証情報を知らないのと同じ構造。

#### セキュリティ対策

| リスク | 対策 |
|--------|------|
| 顧客が大量にAPI呼び出して高額請求 | テナントごとの利用上限（月額上限・1日上限）を設定。超過時は自動停止 |
| 悪意ある利用（APIキーの間接的悪用） | レートリミット + 異常検知。短時間の大量リクエストを自動ブロック |
| FujiTrace側のAPIキー漏洩 | サーバー環境変数に保管、コードにハードコードしない（現行FujiTraceと同じ） |
| 未払い・踏み倒し | プリペイド方式（先にクレジットを購入）またはStripe従量課金（後払い + 上限設定） |

#### 完全自動化への影響

内蔵APIキーにより、導入フローから最大の脱落ポイントが消える:

```
海外競合:
  申し込み → OpenAIアカウント作成 → APIキー発行 → ツールに貼り付け → 設定
  （脱落ポイント: 多い。特に「APIキーって何？」で離脱する層が大きい）

FujiTrace内蔵キー:
  申し込み → ドキュメント登録 → 公開
  （脱落ポイント: ほぼゼロ）
```

**ITリテラシーが高くない日本のSMBにとって、「OpenAIアカウント作成不要」は決定的な差別化。**

#### 着火剤戦略における3つの非対称性（まとめ）

| # | 非対称性 | 競合が追随できない理由 |
|---|---------|-------------------|
| 1 | **価格** — API実費のみ | 競合はチャットbotが収益の柱だから価格を下げられない |
| 2 | **導入の完全自動化** — 人手ゼロ | 競合は営業・CS・エンジニアの人件費が消えない |
| 3 | **APIキー不要** — 内蔵キーモデル | 競合はBYOKモデルのため顧客に技術知識を要求する |

この3つが揃うと: **ITリテラシーが低い企業でも5分でAIチャットbotが本番稼働する。** 日本のどの競合もこれを実現していない。

## 5. 料金設計

### 5.1 チャットbot側（着火剤 = 利益ゼロ）

**表現: 「初期費用0円・月額固定費0円・LLM API従量課金のみ」**

「無料」とは表現しない。日本企業は「無料＝不安」と捉えるため、「従量課金」という形式で安心感を担保する。

| 項目 | 料金 |
|------|------|
| 初期費用 | 0円 |
| 月額固定費 | 0円 |
| LLM API利用料 | 実費（APIプロバイダーの料金をそのまま転嫁） |
| ドキュメント保存 | 一定容量まで無料（超過分は従量課金） |

### 5.2 リアルタイム円換算コスト可視化（2026-03-29追加）

> **FujiTrace本体に組み込む機能。チャットbot専用ではなく、FujiTraceの全ユーザーが恩恵を受ける。**
> ただし、チャットbot着火剤戦略との相乗効果が極めて高い。

#### 背景: なぜ円換算が必要か

OpenAI・Anthropic等のLLM APIは全てドル建て。日本企業にとっての問題:

- 請求書が「$127.43」→ 日本円でいくらか即座にわからない
- 為替変動で月ごとのコストが変わる → 利用量が同じでも金額が変わる → 混乱
- 稟議書・経費精算に「ドル」は書きにくい → 経理部門との摩擦
- **コストが不透明 → 経営層が不安 → AI利用に消極的 → PoC止まり**

これはセクション9.0.4（strategy-2026.md）で定義した「PoC→本番の壁」の一因。

#### 実装方針

**毎日の為替レート（USD/JPY）を自動取得し、全コスト表示を円換算で表示する。**

```
FujiTraceダッシュボード表示例:

┌─ コストサマリー ──────────────────────────────┐
│                                              │
│  本日のコスト:  ¥18,432（$115.20 × ¥160.00）  │
│  今月累計:      ¥287,650                      │
│  先月比:        +12%                          │
│    内訳: 利用量増加 +8% / 円安影響 +4%          │
│                                              │
│  本日の為替レート: ¥160.00/USD（2026-03-29）    │
└──────────────────────────────────────────────┘
```

**重要: 為替変動と利用量変動を分離して表示する。**
「先月より高くなった」が為替のせいなのか利用量のせいなのか、日本企業の経理・経営層はそこを気にする。

#### 技術的な実装

| 項目 | 方針 |
|------|------|
| 為替レートソース | 公開API（例: ExchangeRate-API、Open Exchange Rates等）から毎日自動取得 |
| 取得タイミング | 毎日1回（日本時間朝9時、東京市場オープン時） |
| 保存 | 日次レートをDBに保存（過去レートの参照・コスト再計算に使用） |
| 表示 | ダッシュボードの全コスト表示を円換算で表示（ドル表示も併記） |
| 請求 | 月末締めで、その月の日次レートに基づく円換算額で請求 |

#### ダッシュボードでの表示項目

| 表示項目 | 内容 |
|---------|------|
| リアルタイムコスト | 今日のAPI利用コスト（円換算） |
| 月間累計 | 今月のAPI利用コスト累計（円換算） |
| 先月比（利用量） | 利用量の増減（為替影響を除外） |
| 先月比（為替影響） | 為替変動によるコスト増減 |
| 日次為替レート | 当日のUSD/JPYレート |
| 会話単価 | 1会話あたりの平均コスト（円換算） |
| 予測月額 | 現在のペースでの月末予測コスト（円換算） |

#### 競合との差別化

| | Langfuse | Datadog | LangSmith | **FujiTrace** |
|--|---------|---------|-----------|--------------|
| コスト表示通貨 | USD | USD | USD | **JPY（リアルタイム円換算）** |

**Langfuseも Datadogもやっていない。日本特化の密着軸戦略（strategy-2026.md セクション10.3）の具体的実装。**

#### チャットbot着火剤との相乗効果

チャットbot利用企業にとって、コストの可視化は特に重要:

- チャットbotは会話量が多い（1日数百〜数千件）→ コストが読みにくい
- FujiTraceダッシュボードで「今月のチャットbot運用コスト: ¥48,000」と一目でわかる
- 「1会話あたり¥3.2」という単価がわかれば、経営判断が即座にできる
- **コストの透明性がPoC→本番の壁を突破する最大の武器になる**

### 5.3 FujiTrace側（本業 = 収益源）

既存の料金プラン（`docs/pricing-model.md` 参照）をそのまま適用:

- チャットbot利用者には基本監視を無料で提供（FujiTrace Freeプラン相当）
- 高度な分析・自動改善提案・監査レポート・アラート強化は有料プラン
