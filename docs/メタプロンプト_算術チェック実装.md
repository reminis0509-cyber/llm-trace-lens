## タスク: AI事務員チェック機能のプログラマティック算術チェック実装

### 背景
FujiTrace AI事務員のチェック機能（請求書チェック、見積書チェック等）で、LLM（GPT-4o-mini）が算術ミスを検出できない問題がある。例: 小計600,000の消費税10%が55,000（正しくは60,000）なのに「正しい」と判定してしまう。

既に見積書の専用チェック（`src/routes/tools/estimate-check.ts`）ではNode.jsで算術チェックを実装済み。この同じパターンを汎用チェックタスク（`/api/tools/office-task/execute`のdocument_checkアーキタイプ）にも適用する。

### 目標アーキテクチャ
```
ユーザーが書類テキスト送信
  |
Step 1: LLM(GPT-4o-mini)で書類テキストから数値を構造化JSONに抽出
  -> { items: [{name, quantity, unit_price, amount}], subtotal, tax_rate, tax, total }
  |
Step 2: Node.jsで算術チェック（100%正確）
  -> 各明細: amount === quantity * unit_price
  -> subtotal === sum(items.amount)
  -> tax === floor(subtotal * tax_rate)
  -> total === subtotal + tax
  |
Step 3: LLM(GPT-4o-mini)で形式・内容チェック
  -> 必須記載事項、インボイス番号、日付整合性、表記ゆれ
  |
Step 4: Step 2 + Step 3 の結果を統合して返す
```

### 実装すべきファイル

1. **`src/tools/arithmetic-checker.ts`（新規作成）**
   - 入力: 構造化された金額データ（items配列、小計、消費税、合計）
   - 出力: `{ ok: boolean, issues: Array<{field, severity, message}> }`
   - 既存参考: `src/routes/tools/estimate-check.ts` の算術チェック部分（90行目付近の`performArithmeticCheck`）
   - 消費税率は10%を標準、8%（軽減税率）にも対応
   - 端数処理は切り捨て

2. **`src/prompts/tools/office-task/extract-numbers.md`（新規作成）**
   - LLMに書類テキストから数値を抽出させるためのプロンプト
   - 出力形式: strict JSON
   ```json
   {
     "items": [{"name": "品名", "quantity": 1, "unit_price": 500000, "amount": 500000}],
     "subtotal": 600000,
     "tax_rate": 0.10,
     "tax_amount": 60000,
     "total": 660000,
     "has_financial_data": true
   }
   ```

3. **`src/routes/tools/office-task-execute.ts`（修正）**
   - document_checkアーキタイプの処理フローを変更:
     - Step 1: extractNumbers用プロンプトでLLM呼び出し（数値抽出のみ、GPT-4o-mini）
     - Step 2: arithmetic-checker.tsで算術チェック
     - Step 3: 既存のdocument-check.mdプロンプトでLLM呼び出し（形式チェック、GPT-4o-mini）
     - Step 4: 算術チェック結果をStep 3の結果に統合
   - isCheckTask判定でGPT-4oを使っている部分をGPT-4o-miniに戻す（算術はプログラムでやるので不要）

4. **`packages/dashboard/src/pages/AiClerkChat.tsx`（修正）**
   - チェック結果表示: arithmetic_check結果がある場合はTier分け表示
     - 算術エラー -> 赤色警告（critical）
     - 形式警告 -> 琥珀色警告（warning）
     - 改善提案 -> グレー（info）

### 既存コードの参考箇所
- `src/routes/tools/estimate-check.ts` -- 算術チェックの実装パターン（`performArithmeticCheck`関数）
- `src/prompts/tools/estimate/check.md` -- チェック用プロンプトの書き方
- `src/routes/tools/office-task-execute.ts:112-215` -- アーキタイプ別処理フロー
- `src/routes/tools/_shared.ts:callLlmViaProxy` -- LLM呼び出しヘルパー（OpenAI直接）
- `src/routes/tools/_shared.ts:parseLlmJson` -- LLM出力のJSON安全パース

### 制約
- Node.js (TypeScript ESM)で実装。Pythonではない（プロジェクトはNode.jsスタック）
- 新規npm依存の追加は不要（算術はネイティブJSで十分）
- テストは `src/tests/tools/arithmetic-checker.test.ts` にVitest
- Vercel Serverless Function の60秒制限に注意（LLM 2回呼び出しで合計~15-20秒に収める）
- `enforceFreeQuota(workspaceId, request)` のadminバイパスは維持

### テスト方法
実装後、Chrome MCPで以下をテスト:
1. 正しい請求書 -> チェック結果「ok」
2. 消費税が間違った請求書（55,000、正しくは60,000）-> 算術エラー検出
3. 明細金額が数量x単価と不一致 -> 算術エラー検出
4. 必須記載事項欠落 -> 形式警告
5. UI上で赤/琥珀/グレーの3段階表示確認
