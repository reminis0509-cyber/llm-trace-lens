## 1. エグゼクティブサマリー

FujiTrace は **日本企業向け AI 導入プラットフォーム** です。AI を導入したい中小企業に向けて、実務で使える責任AIツール群・統合された監視UX・枠組み型エージェント（AI事務員）・そしてそれらを下支えする LLM 観測基盤を 1 つの製品に統合します。FujiTrace を使って AI 事務作業を始めるための実用ガイド + ハンズオン教材を全体を包む上位レイヤーに据え、「AI をどう使うか分からない」中小企業を製品自体で育てます。

Mission: **「すべての日本企業に AI を届ける」**

### 顧客向け一行定義

**「監視機能付き AI 事務員プラットフォーム」** — 見積書・請求書等の書類を AI が作成・検証し、全実行が自動で監査ログに記録される。稟議も内部統制も安心。

（※ VC / エンタープライズ向けの「Obs プラットフォーム」「構造的非対称モデル」ナラティブは Section 9 を参照）

### 4つの柱

1. **責任AIツール**（ステージ1 = こどもちゃれんじ） — 見積書 / 請求書 / 納品書 / 発注書 / 送付状 の 5 教材。ボタン一つで作成・検証。
2. **監視**（旧 Watch Room + 履歴 + 統計の統合） — LLM 観測を 1 タブに集約。ambient / リスト / 統計の 3 ビュー切替で全ユーザー層をカバー。
3. **AI事務員**（ステージ2 = 進研ゼミ） — Scaffolded Agent がツール群を組み合わせて事務作業を代行。
4. **実用ガイド**（上位レイヤー） — 「FujiTrace で AI 事務作業を始めるための実用ガイド + ハンズオン教材」。Stripe Docs / HubSpot Academy 型の自社プロダクト中心ドキュメント。3 本柱を包む上位レイヤー。

### 現在地

- **Phase 0（MVP）Week 4** — 2026-04-14 時点
- 既存 5 ツール + AI 事務員カード式 UI + Watch Room W0 + ブログ基盤 + LP デモフォーム + オンボーディング Step 1/2 は **実装済み**
- 次の大型マイルストーン: **Phase A1 開始 = 2026-06-07**（AI Agent MVP、実用ガイド着手、進級トリガー、Team プラン課金）

### 収益モデル（概要・顧客視点）— Claude 方式 3 プラン（2026-04-16 確定）

- **Free（¥0）**: 5時間ごとにリセット（約10回/窓）。自由対話 + function calling で Claude の能力を 100% 活用 + 事務ツール追加
- **Pro ¥3,000/月**: 無制限 + 自律モード + クエスト教材。個人・フリーランス向け
- **Max ¥15,000/月**: Pro 全機能 + 拡張保持期間 + 全文検索 + 優先サポート。チーム・企業向け

価格アンカー:
- **Pro ¥3,000/月 は freee / マネーフォワード帯（¥980〜¥4,980）と同価格帯**。AI 事務員の入口として心理的ハードルを最小化
- **Max ¥15,000/月** は派遣事務員月給の約 1/15。人件費予算から捻出できるため稟議が通る

差別化軸の本質:
- Pro と Max の差は **「個人運用 vs 組織・パワーユーザー運用」**
- Max の本当の価値は **拡張保持期間 + 全文検索 + 優先サポート**
- AI 事務員アーキテクチャは **自由対話 + function calling（Claude 方式）**。制約による弱体化を禁止し、Claude の能力 100% + 事務ツール追加が差別化

裏側の収益戦略（VC / エンタープライズ向け詳細は Section 9）: AI エージェントの実行ごとに trace が自動蓄積され、データ / Obs プラットフォームとしての将来価値を構築する二層構造。

## 2. 上位ポジショニング — FujiTrace 実用ガイド + ハンズオン教材

### 核心メッセージ

**FujiTrace = 日本中小企業が AI 事務作業を始めるための「実用ガイド + ハンズオン教材 + 実務ツール + 観測基盤」**

```
┌──────────────────────────────────────────────────────┐
│  上位レイヤー: FujiTrace 実用ガイド + ハンズオン教材  │
│  (Stripe Docs / HubSpot Academy 型の自社中心ドキュメント) │
├──────────────────────────────────────────────────────┤
│  柱1: 責任AIツール (ステージ1)                        │
│  柱2: 監視 (Watch Room + 履歴 + 統計の統合)          │
│  柱3: AI事務員 (ステージ2・Scaffolded Agent)          │
└──────────────────────────────────────────────────────┘
```

### 正しい比喩: Stripe Docs 型 / HubSpot Academy 型（MDN 型ではない）

ドキュメント・教材プラットフォームには 2 種類ある:

| 型 | 例 | 性格 | FujiTrace との関係 |
|----|----|------|---------------------|
| **MDN 型 / Python docs 型** | MDN / Python docs / W3C | ベンダー中立、言語仕様、技術者向け | **採用しない**（中小企業の経理は読まない） |
| **Stripe Docs 型 / Academy 型** | Stripe Docs / HubSpot Academy / Salesforce Trailhead | 自社プロダクト中心、How-to + ベストプラクティス、利用者向け | **採用**（FujiTrace を使うための実用書） |

FujiTrace の実用ガイドは **「FujiTrace を使って」事務作業を始めるためのもの**。AI 一般論ではなく、製品活用の文脈でだけ書く。

### 採用範囲（教育事業化と明確に区別）

| 解釈 | 内容 | 判断 |
|------|------|------|
| **A. 教育事業化** | カリキュラム販売・スクール運営・コミュニティ運営・講師業 | **絶対不採用**（別事業・別能力・別収益モデル） |
| **B. 教育 = マーケ** | 経営ハッカー型オウンドメディア（SEO） | 既存戦略として継続 |
| **C. 製品内ハンズオン** | 各ツール内に「これは何 / いつ使う / よくある失敗 / 次のレッスン」 | **採用**（柱1〜3に統合） |
| **D. 自社プロダクト実用ガイド** | docs.fujitrace.jp 型、FujiTrace で AI 事務作業を始めるための How-to | **採用**（柱4） |

### コンテンツ設計の原則

| やる | やらない |
|------|---------|
| 「FujiTrace で見積書を作る方法」 | 「見積書とは何か」（一般論） |
| 「AI で請求業務を効率化する 3 パターン」（FujiTrace使用例付き） | 「AI 入門」（FujiTrace 不在） |
| 「FujiTrace で書類成立性を担保する」 | 「契約書法律入門」 |
| 「税理士が FujiTrace を顧問先に勧める時のトーク例」 | 「税理士のための AI 研修」 |

**判断基準**: その記事を読んだ後、ユーザーが「FujiTrace を使う動機」を持つか? Yes なら書く、No ならボツ。

### Why: この上位レイヤーが強い 5 つの理由

1. 中小企業の「AI わからない」問題を製品自体で直接解決する
2. 実用ガイドが SEO 資産となり CAC を下げる（広告依存から脱却）
3. VC ナラティブ強化: 「事務員プラットフォーム + 観測基盤 + 自社実用ガイド」の 3 層
4. 模倣困難 — コードは複製できるが「自社プロダクトを使った実用知識の蓄積」は時間でしか積めない
5. 過去 GTM 失敗の真因「教材不足」を構造的に解消する（代理店 / 直販 / 着火剤すべて使い方を伝える教材が無かった）

### ガードレール（教育事業化への滑り落ちを防ぐ）

- 「教育事業化」「カリキュラム販売」「講座運営」「スクール」「コミュニティ運営」「講師業」の語彙が出たら即ブレーキ
- 教材は **必ず FujiTrace を使う文脈で書く**（製品中心、AI 一般論を独立した教材にしない）
- 「修了証」「ライセンス認定」「資格」を発行しない（教育機関化の入口）
- コンテンツ単体での課金をしない（教材は無料公開、収益はあくまで Pro / Team プラン）
- 競合プロダクトには誠実に触れる（権威性の源、ただし FujiTrace を比較表で勝たせる）
- コンテンツ制作は Founder 1 人前提のため **量より権威性**。1 本を深く、隔週 1 本

詳細: `positioning_education_reference.md`

## 2.5 競合とポジショニング

### 直接競合軸

| 競合 | 戦場 | FujiTrace の非対称性 | 勝ち筋 |
|------|------|---------------------|--------|
| **freee / マネーフォワード** | 会計 SaaS、記帳・請求・確定申告 | 「判断・監査・非構造化・従量・理想主義」の 5 軸で正面衝突しない。周辺業務（見積・納品・発注・送付）と "AI による判断支援" を取る | 会計の代替ではなく、**会計周辺の書類業務を AI で置換**する位置取り |
| **Claude Code / Operator / Devin / Manus** | 汎用エージェント | 自由度が高すぎて日本中小企業の稟議・監査に乗らない。Scaffolded Agent + 日本語 PII + ドメスティックデータで参入障壁 | **日本中小企業市場で「動作範囲が明確、説明責任が取れる」唯一のエージェント**になる |
| **Langfuse / Helicone / PostHog** | LLM Obs | 海外勢は日本語 PII パターンなし、ドメスティック保管なし、日本語 UI なし。GAO Inc との JP パートナー（Langfuse）はいるが local-first ではない | **日本語 Obs の第一想起**を取る |
| **Chatwork / LINE WORKS** | 国内業務コミュニケーション | 直接競合しない。将来は通知 / 承認フローの統合先として協調関係になりうる | 競合視しない |

### 競合が真似できない構造的モート（時間資産）

1. **日本語 PII 検出パターン 15+**: 氏名/住所/マイナンバー/在留カード/運転免許等の日本固有パターンは英語圏 Obs プロダクトに実装されない
2. **日本商慣習の書類成立性チェック**: 見積書の「有効期限」「支払条件」「押印箇所」等、商法・業界慣習への対応。海外プロダクトには無理
3. **ドメスティックデータ保管**: 国内サーバー固定、海外転送なし、日本の個情法・GDPR 互換。エンタープライズ稟議の必須条件
4. **Scaffolded Agent の責任 3 段階グラデーション**: high / medium / low を `responsibilityLevel` で持ち、承認ゲートを設計実装する。汎用エージェントには存在しない
5. **教育リファレンス + ハンズオン**: 中小企業向け AI 導入教材は時間でしか積めない資産
6. **Trace 自動蓄積からのデータフライホイール**: AI 事務員が使われるほど日本企業事務パターンの独占データが蓄積される（Section 9 参照）

### 非対称ポジション図

```
          構造化 / 現実主義
               │
     freee ●─────┼─────
               │
  記録・月額固定 │ 判断・従量・監査
               │
  ─────────────┼─────────── FujiTrace ●
               │
               │ 非構造化 / 理想主義
               ▼
```

freee は左上（構造化・記録）、FujiTrace は右下（非構造化・判断）。5 軸（判断 vs 記録 / 監査 vs 手続き / 非構造化 vs 構造化 / 従量 vs 月額固定 / 理想主義 vs 現実主義）で対称の関係。**顧客は freee を捨てずに FujiTrace も使う**のが期待する導入形態。

### なぜ今か（タイミング論）

- 2025-2026 は LLM コストが **1/10 に低下**した最初の年。AI 活用の現実コストが実務範囲に入った
- 一方、日本中小企業の **90% 以上が AI 未導入**（経産省 DX レポート）
- freee / マネーフォワードが会計領域で先行、**会計周辺業務は空白地帯**
- グローバル AI エージェント各社は日本市場を後回しにしている（言語 / 商慣習 / データ主権の壁）
- **空白の期間は 12-24 ヶ月**。この窓で日本第一想起を取る

## 3. 4つの柱

### 柱1. 責任AIツール（ステージ1・磨き込み中）

**一言定義:** AIが間違えたら実害が出る（金銭・信用・法的）事務書類だけを対象にした、ボタン一つの作成・検証ツール群。

**顧客課題:** 見積書・請求書といった日常書類を、freee 的な「入力→計算→確認」の型で高速生成し、誤字・書類成立性・相場乖離まで自動チェックしたい。チャット画面で白紙から書かせるのは学習コストが高すぎる。

**現状（実装済み）:**
- 5ツール: 見積書 / 請求書 / 納品書 / 発注書 / 送付状
- フォームファーストUX（チャット起点は廃止）
- Layer 1: アプリ内 LLM-as-Judge による品質チェック（3段階 Tier 表示、「FujiTrace 品質チェック」見出し）
- Layer 2: FujiTrace 観測基盤記録（Pro 訴求の核）
- プログラマティック算術チェック（`src/tools/arithmetic-checker.ts`）
- PDF はクライアント側 `@react-pdf/renderer` + Noto Sans JP（サーバー生成は恒久廃止）
- Free（5時間ごとにリセット）、ツール共通ファネルイベント計測

**課金モデル:**
- Free: 5時間ごとにリセット（約10回/窓）、直近 7 日履歴、Watch Room 簡易版
- **Pro ¥3,000/月**: 無制限 + 自律モード + クエスト教材
- **Max ¥15,000/月**: Pro 全機能 + 拡張保持期間 + 全文検索 + 優先サポート

### 柱2. 監視（Watch Room + 履歴 + 統計の統合 / W0 実装済み）

**一言定義:** trace データを 1 つのタブに集約し、ambient / リスト / 統計の 3 ビューに切替できる統合監視レイヤー。FujiTrace のコア UX。

**統合の理由（2026-04-14 確定）:**
- データソースが同じ（trace）= 同じ情報を 2-3 タブに分けるのは技術負債
- 顧客の認知負荷を下げる（「監視タブを開けば全部ある」）
- 開発リソースの分散を防ぐ（1 人で複数 UI を磨き続けない）
- ダッシュボード簡素化（Section 6 参照、5 タブ上限の前提）

**顧客課題:** Datadog / Langfuse / Grafana は問題が起きた時しか開かれない構造的 SaaS 弱点を持つ。trace 監視は退屈で継続されず、企業内の「AI 監視員」ポジションが生まれない。一方、リスト型の履歴画面は「あとから検索」用途では便利だが、毎日見たくはならない。両者を統合し、ユーザーの目的別にビューを切替できる構造にする。

**統合後の構造:**
```
監視タブを開く
 ├─ ambient ビュー (Watch Room 風、リアルタイム流れる、エバンジェリスト向け)
 ├─ リスト ビュー (履歴風、静的、検索フィルタ強、実務担当者向け)
 └─ 統計 ビュー (集計グラフ、コスト推移、エラー率、経営者向け)
```

**現状（Phase W0 実装済み、2026-04-10 本番反映）:**
- 本番 URL: `https://www.fujitrace.jp/dashboard/watch?demo=1`（**デモモードはログイン不要**）
- LINE / Slack 風チャットスタイル積み重ね、時刻カード外側左下（LINE 構図）
- Web Audio API 合成音（PASS / WARN / FAIL / BLOCK × 4 音色、スロットリング）
- ambient warm-white シフト、FAIL/BLOCK 持続グロー、スマート detached 追尾
- LP / Dashboard 整合 Light Theme（ダーク基調はボツ）
- 実装: `packages/dashboard/src/pages/WatchRoom.tsx` 他
- **未完了**: 履歴・統計タブとの UI 統合（Section 6 のダッシュボード再編タスクで実施）

**課金モデル:** Free = 簡易版 / Pro = フル機能 / Max = フル機能 + 拡張保持期間 + 全文検索 + 優先サポート

**Phase W0 完了条件:** Founder のフルスクリーン画面録画 X 投稿でインプレッション 10,000+ → Phase W1 着手判断。

**詳細:** `vision_watch_room.md` / `watch_room_w0_complete.md`

### 柱3. AI事務員（ステージ2・Phase A1 で本格実装）

**一言定義:** **自由対話 + function calling（Claude 方式）AI Clerk** — Claude の能力を 100% 活かし、事務ツールを function calling で追加する設計。制約による弱体化を禁止。「AI事務員」としてエンドユーザーに提示する。

**顧客課題:** 汎用エージェント（Claude Code / Operator / Devin / Manus 等）は事務ツールを持たない。一方、制約型パイプラインで能力を絞ると「使い物にならない」。Claude 方式（自由対話 + function calling）で AI の全能力 + 事務特化ツールの両取りをする。

**アーキテクチャ（2026-04-16 更新 — Claude 方式に移行）:**

```
┌─ Start: 入力契約（最小限）────────────────┐
│ ベースライン: 基本会社情報（初回 setup 済）│
│ タスク毎: 最小限の構造化情報              │
│ 不足時: AI が「○○が必要です」と促す     │
│ ファイルアップロード可（PDF/Excel/画像）  │
└─────────────────────────────────────────┘
                 ↓
┌─ Middle: 実行自由ゾーン ─────────────────┐
│ ✅ LLM 推論・チェーン思考                 │
│ ✅ テキスト処理（要約/翻訳/整形/リライト）│
│ ✅ 数値計算・算術検証                     │
│ ✅ ワークスペース内データ読取             │
│ ✅ ユーザーへの質問（対話的）             │
│ ✅ 既存書類テンプレ活用                   │
│ ❌ 外部 API / HTTP / シェル / ブラウザ    │
│ ❌ 金融取引・支払実行                     │
│ ❌ 第三者への送信（メール送信等）         │
└─────────────────────────────────────────┘
                 ↓
┌─ End: 出力契約（固定形式）───────────────┐
│ PDF（5書類）/ Excel / CSV / 構造化テキスト│
│ 全出力は validation を通過                │
│ 送信・確定は必ず人間承認ゲート経由        │
└─────────────────────────────────────────┘
```

**入力契約の原則（Founder 指示 2026-04-14）:**
- **絶対必要な基本情報は初回 setup で一度だけ**（既に AI事務員で実装済みの会社基本情報モーダルを活用）
- **タスク毎の入力は最小限**に抑える
- **不足情報があれば AI が対話的に促す**（「○○の情報が不足しています、教えてください」）
- 空白プロンプト問題は「入力契約の明示」で自然に消える

**中間自由ゾーンの境界（厳密）:**
- 自由: ワークスペース内読取・計算・推論・テキスト処理・ユーザー質問
- 禁止: 外部ネットワーク・シェル・金融取引・第三者送信
- 承認ゲート経由で可: メール下書きの送信・PDF の正式発行・外部システム連携（Phase A2 以降）

**なぜ Contract-Based 設計が優れているか:**
1. 顧客説明が1行で済む: 「必ず PDF/Excel が出ます、途中で変なことはしません」
2. 審査が Input/Output だけ = 監査が簡潔
3. AI 進化をそのまま享受（契約維持で内部は勝手に賢くなる）
4. 「事務用」保証が出力形式から自然に導出
5. 実装が軽い（ツール enumeration 不要）
6. 競合との差別化が鮮明:
   - Claude Code / Operator: 契約なし、全部自由（エンタープライズ困る）
   - freee: 入力も中間も出力も硬い（AI の創造性を発揮できない）
   - **FujiTrace: 入口と出口だけ固定、中間は知的**（両取り）

**UI とチュートリアル段階:**
- カード入口（既存・残す）: 固定 Input フォーム → 固定 Output (PDF)
- チャット入口（Phase A1 追加）: 自由 Input → AI が質問・処理 → 固定 Output
- チュートリアル 4 Level: フォーム厳密 → フォーム+自由メモ → 自由+AI質問 → 完全自由（熟練者）

## 7. やらないこと・廃止戦略

箇条書きで理由を明示。「やらない」を明文化することで迷走を防ぐ。

- **教育事業化（解釈 A）** — 別事業・別能力・別収益モデル。採用は C + D のみ。以下の語彙が出たら即ブレーキ:
  - カリキュラム販売 / スクール運営 / 講座運営 / 講師業 / 教育事業
  - コミュニティ運営 / オンラインサロン / 会員制学習プラットフォーム
  - 修了証発行 / ライセンス認定 / 資格付与 / バッジ発行
  - コンテンツ単体課金 / 教材販売 / 有料記事
  教材は無料公開、収益はあくまで Pro / Team プランから。「FujiTrace を使う動機」を持たせる文脈でしか書かない。
- **自律エージェントの scaffold 外し** — 責任 AI ツール戦略と両立しない。Claude Code / Operator 型の全自由エージェントは FujiTrace では作らない。
- **モバイルアプリ** — PMF 前は不要。freee ですらスマホは確定申告特化。業務ツールは PC 前提が業界共通認識。再検討は PMF + ユーザー 1,000 超達成後。
- **大規模広告投下** — 過去 3 キャンペーンで ¥20,000 出血済み。広告再開は教育コンテンツ + オーガニック流入が立ち上がった後。広告投下前は **ファネル設計 + Keyword Planner** の両方完了が絶対条件（Founder と約束済み）。
- **押し売り型代理店** — 2026-03-12 に失敗証明済み。「商材＝教材」が完成するまで再開しない。
- **プレスリリース** — 無料 PR 配信網はすでに機能していない。
- **士業独占業務領域** — 確定申告（税理士法）/ 会社設立・許認可（行政書士法 / 司法書士法）/ 就業規則・社保（社労士法）/ 契約書法的助言（弁護士法 72 条）/ 投資助言（金商法）/ 診断行為（医師法）/ マイナンバー代行（個情法）は絶対不可。
- **チャット起点 UI の復活** — 2026-04-09 E2E で Founder が使わなかった。責任 AI ツールはフォームファースト固定。
- **サーバーサイド PDF 生成** — Vercel 4.5MB 制限 + pdf-lib CJK subset バグで 2 回失敗済み。全ツール恒久的にクライアント側 `@react-pdf/renderer`。
- **無制限 Free tier** — API 実費が出るため持続不可能。Free は 5 時間ごとにリセット（約 10 回/窓）に固定。
- **Benesse の "進級強制"** — お試し 1 回券は不採用。教育動画で欲しがらせる。

## 9. 付録: VC / エンタープライズ向け追加ナラティブ（2026-04-14 改訂）

2026-04-14 のメッセージング二層化判断により、構造的非対称性・Obs プラットフォーム・データモートのナラティブは顧客向け資料から外し、以下の対象者にのみ提示する。

### 対象と使い分け

| 対象 | 使うナラティブ | 目的 |
|------|--------------|------|
| 顧客 / LP / 営業 | 「監視機能付き AI 事務員プラットフォーム」（Section 1 参照） | シンプル訴求、価格比較で勝つ |
| エンタープライズ / 監査担当 | 「AI 導入時の監査統制基盤が標準搭載」 | コンプラ予算から稟議通過 |
| VC / 投資家 | 本 Section のフルナラティブ | 差別化・モート・評価倍率向上 |
| 開発者 / OSS コミュニティ | 「LLM 観測の日本語標準 + Scaffolded Agent フレームワーク」 | 技術信頼性・エコシステム |

### VC 向けフルナラティブ

**FujiTrace は、AI を原価提供することでデータを獲得し、観測プラットフォームとして収益化する構造的非対称モデルです。**

表層（顧客が見るもの）:
- AI 事務員プラットフォーム
- 監視機能付き
- Free / ¥3,000 / ¥15,000 の分かりやすい 3 プラン

深層（VC が評価するもの）:
- **AI コスト提供 × Obs / データプラットフォーム収益化の二層構造**
- freee (記録/構造化/月額固定) に対する 5 軸非対称ポジション（判断 / 監査 / 非構造化 / 従量 / 理想主義）
- 日本語 PII 検出（15+ パターン）+ ドメスティックデータ保管 + Scaffolded Agent の日本市場参入障壁
- Claude Code / Operator / Devin 等のグローバル汎用エージェントが日本中小企業市場で勝てない構造的理由

### データ / Obs フライホイール

AI 事務員が使われるほど trace が自動蓄積する構造を持つ。Obs を売らなくてもデータは溜まる。

```
AI事務員が普及する
    ↓
trace 大量蓄積（日本企業事務作業パターンの宝庫）
    ↓
 (a) LLM-as-Judge 精度向上 → プロダクト品質優位
 (b) 業界別ベンチマーク公開 → ブランド権威
 (c) B2B2B データ提供（保険 / 与信 / コンプラ） → 二次収益源
 (d) FujiTrace 専用モデル fine-tuning → 技術モート
    ↓
競合が時間でしか追いつけない構造的優位
```

### 評価倍率の二重構造

- 顧客獲得段階: AI エージェント屋さんの評価倍率（10-30× ARR）で ARR を積み上げる
- Exit 段階: データ / Obs プラットフォームの評価倍率（20-50× ARR）で評価される
- この二重構造により、同一 ARR でも Exit バリュエーションが大きく伸びやすい

### 日本 BtoB SaaS 勝ちパターンとの整合

- HubSpot (マーケティングツールを売りながら顧客データプラットフォーム化)
- Salesforce (SFA を売りながら CRM 統合プラットフォーム化)
- freee (会計ソフトを売りながら金融データプラットフォーム化)

FujiTrace も同型: AI 事務員を売りながら、日本企業 AI 活用のデータプラットフォーム化する。

### 禁止事項（メッセージング二層の厳守）

- 顧客向け資料に「構造的非対称」「Obs プラットフォーム」「データモート」の語を出さない
- VC 向け資料に「¥3,000」「freee 代替」等の個別製品訴求を前面に出さない
- Pro ¥3,000 / Max ¥15,000 は両者で共通。ただし文脈は顧客 = 「低コスト AI 事務員」、VC = 「ARR 積み上げアンカー」

詳細メモリ: `pro_bundle_pivot.md` / `agent_billing_model.md`

## 12.14 AI社員構想 — Manus型TTPと日本業務ローカライズの二軸戦略（2026-04-20 確定）

### エグゼクティブサマリー

2026-04-20、Founder は FujiTrace の「AI 事務員」を「**AI 社員**」へ格上げする意思決定を行った。同時に、Manus AI（Butterfly Effect 社、Meta が 2025-12 に US$2-3B で買収） [1][3] の Projects アーキテクチャを **TTP（徹底的にパクる）** 対象として採用し、日本業務特化での再実装方針を確定した。合わせて料金体系を既存 3 プランから **5 プラン（Free / Pro / Team / Max / Enterprise）** に拡張、実装は Week 単位を廃止し v1 スコープ一括投下とする。本 Section は Section 12.11 / 12.12 / 12.13 で確立した AI 事務員・Scaffolded Agent・Contract-Based Runtime β の延長線上に位置し、既存 Section の記述を上書きせず、上位レイヤーとして包含する構造を取る。

### 12.14.1 「点」から「線」への進化原則

現状の FujiTrace は「書類 1 件 = セッション 1 件 = 点」で完結する設計である。Founder 判断により、FujiTrace を「使って仕事する」ツールではなく、「**FujiTrace の中で仕事が進む**」ワークスペースに格上げする。

| 評価軸 | 「点」状態（現状） | 「線」状態（AI 社員 v1） |
|-------|-------------------|-----------------------|
| 作業単位 | 書類 1 件ごとにセッション終了 | 業務全体がタイムライン化 |
| コンテキスト | セッション内のみ | ワークスペース永続記憶 |
| ユーザー行動 | 必要な時に立ち上げる | 朝開いたら今日の予定が並ぶ |
| アナロジー | 単機能 Web ツール | Claude Code（フォルダに物語を作る） |

**キーアナロジー:** 「Claude Code で仕事をしている」状態と「仕事に Claude Code を使っている」状態は別物である。前者でなければツールに成り下がる。Claude Code がフォルダの中に物語を作るように、FujiTrace はユーザーのワークスペース内に業務の物語を作る。新機能設計は常に「**線化に貢献するか?**」で問い返す。ツールに留まる設計は却下。

### 12.14.2 命名: AI 事務員 → AI 社員

「事務員」メタファーは Section 12.11 で空白プロンプト問題を解決した戦略資産であるが、器（対象スコープ）を 5 書類から業務全般へ拡張するにあたり、外部呼称を格上げする。

| 観点 | AI 事務員（旧呼称） | AI 社員（新呼称） |
|------|-----------------|----------------|
| 対象スコープ | 書類業務 5 種 + 132 項目カタログ | 業務全体の線（予定 / 連絡 / 書類 / 集計 / 将来は会議参加） |
| Scaffold の深さ | 単発タスクのメタファー | 朝来て業務を引き継ぐ社員のメタファー |
| 顧客向けメッセージ | 「事務員 1 人採用」 | 「AI 社員を雇う」 |
| 価格アンカー | 派遣事務員月給の 1/15 | 正社員給与帯との対比に拡張可 |

- **外部名称（LP / Dashboard / meta タグ / docs）:** 「AI 社員」へ完全移行
- **内部コード識別子:** `ai-clerk` / `AiClerk*` / `office-tasks/` 等は工数節約のため維持
- 既存 Section 12.11 で定義した「事務員」メタファーは AI 社員内側の実装詳細として引き続き有効（否定・削除しない）

### 12.14.3 Manus AI TTP 分析

Manus AI（Butterfly Effect Pte Ltd、Singapore） [1] は 2025-03-06 ローンチの自律型エージェント製品で、2025-12 に Meta が US$2-3B で買収した [3]。招待制ベータに 200 万人超の待機リストを持つ [1]。FujiTrace AI 社員 v1 の設計基準として、Manus の採用要素 / 不採用要素を明示する。

| Manus 要素 | 内容 | FujiTrace での扱い |
|-----------|-----|-----------------|
| Projects（永続ワークスペース） [4] | 指示 / 参照ファイル / 会話履歴 / 接続 Connector をセッションを跨いで保持 | **TTP 採用** → `workspace_memory` + `task_timeline` として実装 |
| MCP + OAuth Connector [5] | Gmail / Calendar / Drive / Notion / Stripe / Slack / HubSpot / HF / GitHub の 9 種 | **TTP 採用** → `src/connectors/` + `connector_tokens` テーブル、ただし日本 SaaS 優先 |
| Plan / Execute / Verify マルチエージェント | 3 役で協調、Claude 3.5 Sonnet + fine-tuned Qwen のハイブリッド | **既存 Contract-β 拡張**（Section 12.13 の 4528207 実装を延長、whitelist 8 ツール → 拡張） |
| Manus's Computer（作業ライブビュー） | ブラウザ操作 / 検索 / フォーム入力を可視化 | **既存 SkeletonTrace 拡張** — 2026-04-13 実装済み基盤を Connector 操作にも拡大 |
| 2 モード（Chat / Agent） | 会話モード vs 自律多 step 実行 | **既存カード / 自律 β で対応済み** |
| VM sandbox per task（E2B 基盤） [9] | ユーザー毎に独立したブラウザ / ターミナル / ファイルシステム | **Phase A2 以降、v1 では非採用** — 日本企業は専有 VM より国内 SaaS 信頼が強い |
| Zapier 統合 [5] | 数千アプリへの橋渡し | **非採用** — 日本企業の利用率低、freee / サイボウズ優先 |
| クレジット制課金 [2][6] | タスク複雑度依存、失敗時も消費 | **非採用** — 料金透明性で差別化（後述 12.14.9） |

**TTP の必然性:** Manus は Agent Mode の完成度で業界標準を一気に取りに行った。Projects + Connector + Plan/Exec/Verify の 3 点セットを自前で再発明するより、設計済みのアーキテクチャを日本業務に適用した方が到達時間が短い。ただし Meta 買収後の地政学リスク（中国当局調査 [1]）と欧米エンタープライズ警戒を踏まえ、**国内データ滞留 + 日本 SaaS 深連携 + 料金透明性**を非対称軸に据える。

### 12.14.4 3 層アーキテクチャ

AI 社員 v1 は以下の 3 層構造で実装する。既存 Section 12.13 の Contract-Based Runtime β は Runtime 層として位置付け直す。

```
┌─────────────────────────────────────────────────┐
│ Workspace 層 (永続 / Manus Projects 対応)         │ ← 線の実体
│  ├ workspace_memory       (ユーザー指示 / 基本情報)│
│  ├ task_timeline          (業務履歴)              │
│  ├ past_documents         (発行済書類インデックス) │
│  └ customer_db            (取引先マスタ)          │
├─────────────────────────────────────────────────┤
│ Runtime 層 (実行 / Section 12.13 拡張)            │
│  ├ Plan → Execute → Verify Loop                  │
│  ├ Contract-Based 制約 (whitelist tools)         │
│  └ SkeletonTrace ライブビュー                     │
├─────────────────────────────────────────────────┤
│ Connector 層 (I/O / MCP + OAuth)                 │ ← 手足
│  ├ src/connectors/google-calendar                │
│  ├ src/connectors/gmail                          │
│  ├ src/connectors/chatwork  (日本特化)            │
│  ├ src/connectors/freee     (日本特化)            │
│  └ connector_tokens / oauth_states               │
└─────────────────────────────────────────────────┘
```

- **Workspace 層:** AI 社員のメモリ本体。Founder 確定の「基本会社情報モーダル」（Section 12.11 実装済み）を永続化の出発点として拡張
- **Runtime 層:** Section 12.13 で β 着地した Contract-Based Runtime を母体として採用。whitelist ツール群を Connector 層に接続
- **Connector 層:** MCP + OAuth で外部 SaaS に接続。**この層が最大の参入障壁であり、同時に OSS 化しないクラウド独占領域（12.14.10 参照）**

### 12.14.5 Connector 優先順位

Manus の 9 コネクタ構成 [5] をそのまま模倣せず、日本 BtoB 市場の利用実態に合わせて優先順位を再設計する。

| 優先度 | コネクタ | 採用根拠 |
|-------|---------|--------|
| S | Google Calendar | 線化の核、朝のブリーフィング（今日の予定）の必須入力 |
| S | Gmail | 書類送付 = 業務の大半、下書き生成 → 人間承認の鉄板フロー |
| A | Chatwork | 日本中小企業でシェア高、Manus が持たない **非対称軸** |
| A | freee | 会計逆輸入、Manus が持たない、Section 2.5 で定義した 5 軸非対称の強化 |
| B | Google Drive | 書類保管標準、過去書類検索の基盤 |
| B | Slack | 内部連携、開発者コミュニティ流入時に効く |
| C | マネーフォワード | freee 代替、2 強カバーで会計導線完成 |
| C | サイボウズ | 稟議 / 押印フローとの接続、Enterprise 訴求の武器 |

S / A 優先コネクタ 4 種を v1 で実装（Google 2 種は本実装、日本 2 種は「近日対応」UI 枠 + Phase A1 本実装）、B / C は Phase A1 以降の展開枠。Manus が Zapier で横展開するのに対し、FujiTrace は **日本 SaaS を深く縦掘り**する戦略を取る。

### 12.14.6 日本ローカライズ差別化の 6 柱

Manus が Meta 傘下・Singapore 拠点・USD 建て料金 [2] である構造を踏まえ、日本市場で Manus が埋められない 6 つの軸で非対称ポジションを確立する。

1. **日本業務商習慣対応:** 稟議 / 押印 / インボイス制度適格請求書 / 支払サイト（末締め翌月末払い等） / 源泉徴収 / 振込手数料負担ルール。Manus の汎用ワークフローでは対応不能。Section 7.8.4 / 事務作業カタログ v2 付録 D「押印箇所 AI 指摘」が象徴機能。
2. **日本 SaaS 深連携:** freee / マネーフォワード / サイボウズ / Chatwork / LINE WORKS 等、Manus の 9 コネクタには含まれない日本 SaaS を Connector 層で優先実装。日本企業の標準業務システムを直接操作できる AI 社員は現時点で存在しない。
3. **国内データ滞留:** Meta 傘下 Manus に対する明確な非対称軸。個情法対応 + 国内 AWS リージョン + ベクトル DB 国内配置。エンタープライズ稟議通過の必須条件。
4. **料金透明性:** 月額固定 or 回数明示で、Manus のクレジット制 [2]（失敗時も消費・月跨ぎ予測不能 [6]）を意図的に回避。日本企業の購買慣習と整合。
5. **セキュリティ認証ロードマップ:** P マーク → ISO27001 → SOC2 の順で取得。Manus は SOC2 / ISO27001 公開なし、機密入力非推奨との記載あり [6] — これを攻撃面として Enterprise プランで対比する。
6. **堅実性（Write-access 領域で勝つ）:** 書き込み前の人間承認ゲート、バリデーション層の厚み、プログラマティック算術チェック（Section 12.11 実装済み）。Manus の Write-access 崩壊弱点 [6] を構造的に補完する設計。

### 12.14.7 Manus の弱点 7 つ（突破口リスト）

Manus の既知の弱点を FujiTrace の突破口として戦略ドキュメントに明記する [1][6][7]。営業・VC 対話・採用時に参照する。

| # | Manus 弱点 | FujiTrace 突破口 |
|---|----------|----------------|
| 1 | SOC2 / ISO27001 公開なし | Enterprise プランで監査統制基盤バンドル、認証ロードマップ明示 |
| 2 | 幻覚（価格 / 統計 / 引用の捏造） | プログラマティック算術チェック + LLM-as-Judge 二層検証 |
| 3 | "High service load" エラー常態化 | 国内インフラ + Vercel + 低並行性要求の書類業務優先で回避 |
| 4 | ガードレール弱（途中迷走 / CAPTCHA 停止） | Contract-Based Runtime の whitelist 8 ツール + MAX_ITER=5 + 90s timeout（Section 12.13 実装済み） |
| 5 | Write-access 崩壊（本番投入非推奨） | 書き込み前の人間承認ゲート、5 書類の厳密 Output 契約 |
| 6 | 地政学リスク（中国規制調査 + Meta 買収） | 日本法人 + 国内データ滞留 + 日本語ドキュメンテーション |
| 7 | Team 機能浅い（SSO / 共有クレジットのみ、稟議概念なし） | サイボウズ連携 + 承認フロー + 稟議ワークフロー対応（Enterprise） |

### 12.14.8 「禁止された」弱点輸入リスト

TTP といえども輸入しない要素を明示する。既存 Section 1 / Section 12.11 の料金体系・顧客訴求との整合を維持する。

- **クレジット制料金:** Manus 最大の顧客不満点 [2][6] を輸入する理由なし
- **VM sandbox（E2B 依存）:** v1 では非採用、Phase A2 以降で判断
- **Zapier 統合:** 日本利用率低、freee / Chatwork / サイボウズを優先
- **Meta 統合前提の設計:** FujiTrace は独立プラットフォーム、Meta AI 統合はしない

### 12.14.9 料金体系拡張（3 プラン → 5 プラン）

Section 1「収益モデル」で確定した Claude 方式 3 プラン（Free / Pro ¥3,000 / Max ¥15,000）を、AI 社員の器拡張に合わせて **5 プラン体系**に拡張する。既存 3 プランの価格・位置付けは変更しない（Section 12.12 の pricing_claude_model_2026-04-16 を完全維持）。

| プラン | 月額 | 新規 / 既存 | 位置付け |
|-------|-----|-----------|--------|
| Free | ¥0 | 既存維持 | 5 時間ごとリセット、Phase 0 の獲得入口 |
| Pro | ¥3,000 | 既存維持 | 個人 / SOHO、freee 帯価格アンカー |
| **Team** | **¥6,000/seat（最低 2 席 = ¥12,000〜）** | **新設** | 中小企業向け、共有ワークスペース + チーム分析 |
| Max | ¥15,000 | 既存維持 | パワーユーザー、拡張保持期間 + 全文検索 |
| **Enterprise** | **¥50,000〜（年契約）** | **新設** | 大規模 / 稟議対応、SSO / 承認フロー / 監査統制 |

**クレジット制は意図的不採用:** Manus Team プランは $40/seat（最低 2 席）+ 4,000 クレジット/seat + 共有クレジット制 [2]。FujiTrace Team は **回数制限なし + 月額固定**で、中小企業の購買予算編成と整合する。Lindy 等も含めた業界分析 [8] でクレジット制は顧客側予測困難性が普遍的不満点であるため、非対称軸として「月額固定」を戦略的に選択する。

**価格アンカー:**
- Team ¥6,000/seat = 派遣事務員日給の約 1/50、複数人利用で稟議通過しやすいレンジ
- Enterprise ¥50,000〜 = 監査統制バンドル含め freee エンタープライズ帯と同レンジ

### 12.14.10 OSS 戦略 tie-in

`decision_oss_scope_2026-04-20.md` で確定した **Langfuse 型 2 層 OSS モデル**を AI 社員 3 層アーキテクチャに当てはめる。

| 3 層 | OSS / クラウド | 根拠 |
|------|-------------|----|
| Workspace 層（永続メモリ） | **OSS 化** | コミュニティ獲得 + 採用チャネル、Langfuse 先例で市場リーダーに |
| Runtime 層（Contract-β 拡張） | **OSS 化** | Plan / Exec / Verify loop は汎用技術、公開で技術信頼を獲得 |
| Connector 層（日本 SaaS 含む） | **クラウド独占** | OAuth 審査 / セキュリティ監査 / API 追従の運用コスト高、OSS 化すると品質維持困難 |
| 日本ビジネス知識（132 項目カタログ / 書類テンプレ / 商習慣） | **クラウド独占** | 最大の参入障壁、Manus 含む競合が真似しづらい資産 |

**公開順序:** AI 社員 v1 実装完了 → `fujitrace-core` を Apache 2.0 or MIT で OSS 公開 → 採用チャネル化。Commercial addon 禁止条項は入れない（Manus / Langfuse 流）。
