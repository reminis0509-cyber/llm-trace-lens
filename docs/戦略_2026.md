## 1. エグゼクティブサマリー

### Mission

**「すべての日本企業に AI を届ける」**

### 顧客向け一行定義(LP / 営業 / LINE で使う言葉)

> **採用市場が壊れた日本で、人を雇わずに机上作業を AI 社員に任せられるサービス。**

### VC / 投資家向け一行定義(深層ナラティブ)

> **AI 事務員を入口に、日本企業事務作業の trace データを独占蓄積するデータ / Obs プラットフォーム。**

### 差別化の核(本書で確定)

| 軸 | 内容 |
|---|---|
| **業務スコープ** | **机上作業に特化**(製造業の工場系 / 人事健康管理は明示的に外す) |
| **手段** | AI 社員(Scaffolded Agent)+ 5 書類ツール + Watch Room 監視 + 教育リファレンス |
| **チャネル** | LINE Gateway AI(身近な相談相手)→ Web fujitrace.jp(本格作業) |
| **データ** | trace 全件永続記録(2026-04-26 バケツ穴修復で初めて完全成立) |

### 現在地(2026-04-27)

- **Phase 0 (MVP) Week 6** — 全機能が本番反映済み
- 既存実装: 5 書類ツール / AI 社員カード式 UI / Watch Room W0 / 4 章チュートリアル + 修了証 / ブログ基盤 / LP デモフォーム / LINE Phase A1-A5(Welcome Journey + Rich Menu 6 機能 + キャラ刷新 + 内蔵 Web 検索)/ Trace recorder(バケツ穴修復済)
- **次の大型マイルストーン**: Phase A1 開始 = 2026-06-07(Team プラン課金 / AI Agent 正式版 / 教育レイヤー起動)

### 数字目標(Phase 0 北極星)

- **有料契約 3 社**(Pro + Team 合算、2026-06-06 まで)
- 失格条件: 有料 0 社のまま 2026-06-06 到達 → 価格・訴求・プロダクトの根本見直し

### コアストーリー(LP / 営業ピッチ / VC 1on1 共通)

中小企業の社長 / 社員 / 事業の 3 主体に、おしごと AI(+ 隣にカピぶちょー)を導入することで起こる時系列の変化を 1 ページに圧縮した「明るい未来」テンプレート。本書 Section 9.5 で詳述する物語の骨格。

```
[現状の中小企業]
  社長:  雑務に潰れて経営判断できない、子どもの顔が見られない
  社員:  単純作業の繰り返しで成長実感ゼロ、人手不足で疲弊
  事業:  採用できない・辞める・業績伸びない

       ↓  おしごと AI 導入(¥3,000 ちょい〜、月給 1/15 の AI 社員)
       ↓  + カピぶちょー(関西弁の上司、隣のデスクで励ます)

[3 ヶ月後]
  社長:  雑務 8 割を AI に → 経営判断と新規事業に時間を使える、早く家に帰れる
  社員:  AI を部下にし、自分は判断と顧客対応に専念 → スキルが上がる、成長実感
  事業:  採用しなくても回る、離職率低下、業績向上
  オフィス: カピぶちょーが「お疲れさん〜」と励ます、和やかな空気

       ↓  さらに 3 年後

[長期未来]
  社長:  「うちの会社、面白いことやってる」と社員が胸を張る会社に
  社員:  AI と共に働くスキルを身に付け、市場価値が上がった
  事業:  中堅企業相手にも「なんか違うな」と思わせる会社になった
```

中核命題: **「AI でルーチンワークを殺して、人間が人間らしい仕事だけをする」**(Founder + Grok ペルソナ + 本書第 4 章「明るい未来」が独立に同じ結論)。

キャラ構造: **AI が実務、カピぶちょーが空気を作る** — 現実のオフィスで「実務担当の若手 + 関西弁の上司」が同居する構図を画面内で再現する(Section 7.3 詳述)。

## 2. ビジネスの本質

### 2.1 ニーズ(顧客の真の痛み)

**旧定義(2026-04-14 までの誤り)**: 「人を雇う余裕がない」 = **金欠ペイン**

**新定義(2026-04-26 Founder 修正)**: **採用市場が壊れて人手不足が解決できない** = **採用構造ペイン**

なぜ修正したか:
- 「金欠」は零細(1-5 人企業)限定で薄利確定 → Pro ¥3,000 帯しか取れない
- 「採用できない」は**規模問わず全層**に存在(中小企業庁 2024:「人手不足が経営圧迫」回答 67%、零細 70% / 小 65% / 中 60% / 中堅 55%)
- 採用構造ペインは **Team / Max / Enterprise の seat 単位課金**まで通る = M&A 価値の根幹となる厚利層を含む

### 2.2 ウォンツ(顧客が口にする手段)

**「中小企業 DX」「AI 導入したい」** = 抽象的な手段カテゴリ。これ自体は便宜的なフックに過ぎない。

ウォンツとニーズの違い:
- ウォンツ = 顧客が「これが欲しい」と言う言葉(中小企業 DX、業務効率化)
- ニーズ = その背後にある真の問題(採用できない、社長が雑務に潰れる、子育て介護の時間が無い)

**LP / 営業のコピー設計原則**: **ウォンツの言葉で集客、ニーズの言葉で刺す。**

### 2.3 商品 = 問題解決の道具

第 2 章で本が言っているとおり「**商品は問題解決のための道具にすぎない**」。

FujiTrace が提供する**最高の未来**(本書第 4 章「明るい未来とセットで提示」):

> 3 年後の御社の朝 8:30。社長は AI のダッシュボードを 10 分で承認、営業は 3 社訪問できる、経理はゼロタッチ。残業ゼロ、離職率ほぼゼロ。社員は「自分たちは AI の上司」と胸を張る。

(Grok ペルソナ社長像から抽出 / 2026-04-26)

**売上は上がらない**。**時間 = コスト = 利益**を返す。社長の深層欲求:
- 自分の時間(早く帰る / 子育て介護 / 趣味)
- 経営に集中したい(雑務から解放)
- 「なんとかなる」安心感(人を増やせない不安からの解放)

### 2.4 1 行に圧縮

> **FujiTrace = 採用できない日本企業に、月 ¥3,000 から AI 社員を貸し出す机上作業特化サービス。**

入口は LINE で身近な相談相手として始まり、出口は Web で 5 名分の AI 社員チームを動かすところまで階段を上る。

### 2.5 人間らしい仕事への回帰(価値訴求の心臓)

**FujiTrace の存在意義は「コスト削減」ではなく「人間が人間らしく働ける状態への回帰」である。** これは戦略全体の通奏低音として、各柱・各機能・各コピーが収束すべき中核メッセージ。

#### 3 主体での価値構造

| 主体 | AI 導入前 | AI 導入後 |
|---|---|---|
| **社長** | 見積書 / 請求書 / 経費 / メールで 1 日が終わる、経営判断ができない、家に帰れない、子育て介護の時間ゼロ | 雑務の 8 割を AI に。経営判断 / 営業 / 新規事業に時間を全振り。早く家に帰れる |
| **社員** | 単純作業の繰り返し、成長実感ゼロ、自分の市場価値が下がる不安 | AI を「優秀な新入社員」として上司の立場で使いこなす。判断 / 顧客対応 / 提案に専念。**AI と共に働くスキル**で市場価値が上がる |
| **事業** | 採用できない、採れても辞める、人手不足で業績伸びない | 採用に依存しない事業設計、社員の離職率低下、人間ならではの付加価値で他社差別化 |

#### 「コスト削減」訴求は二の次

派遣事務員月給 ¥25 万に対し AI 社員 Max ¥15,000 = 1/15 のコスト効率は本書 Section 8 のアンカーとして機能するが、これは**入口の説得材料**であり**真の価値ではない**。真の価値は「**社長の時間が返り、社員が成長し、事業が前に進む**」という 3 主体共通の明るい未来。

本書第 4 章「商品はあくまで問題解決の道具」の通り、商品(AI 社員)は手段、目的(人間らしい仕事)を売る。

#### 既存戦略要素との統合

「人間らしい仕事への回帰」が中核メッセージとして機能することで、これまで分散していた戦略要素が **1 つの物語の登場人物** として統合される:

| 既存要素 | 「人間らしい仕事」ストーリーでの役割 |
|---|---|
| AI 社員(柱 3) | 雑務を肩代わりする働き手、人間の上司の指示で動く |
| **カピぶちょー(Section 7.3)** | **AI に教える上司キャラ、ユーザーに励ましと共感を与える、オフィスの和やかな空気を作る** |
| 4 章チュートリアル + 修了証(柱 4) | 社員の AI スキル成長 = 市場価値上昇の証(チュートリアルはカピぶちょーが解説) |
| Watch Room(柱 2) | 「AI に任せても安心」を保証する見えない基盤(異常検知時はカピぶちょーが ambient で通知) |
| ゲーミフィケーション(Section 5.5) | 「AI を育てる」「使いこなす」実感装置(レベルアップ時にカピぶちょーが祝福) |
| LINE Welcome Journey | 身近な相談相手として、AI と人間の関係性を育てる起点(Step 3 雑談はカピぶちょーが主役) |
| 教育リファレンス(柱 4) | AI 時代の働き方を学ぶ場(カピぶちょーが先生役) |

## 3. ターゲット階層(誰に売るのか)

### 3.1 規模階層 × 支払い能力 × LTV

| 層 | 規模 | 状況 | 適合プラン | 月額/seat × 平均 seat | LTV(24ヶ月想定) |
|---|---|---|---|---|---|
| A. 零細 | 1-5 人 | 雇いたくても無理、ITリテラシー低 | **Pro** | ¥3,000 × 1 | ¥10K-30K(解約率高) |
| **B. 小企業** | **5-30 人** | **事務員 1-2 名いるが採用追加できない** | **Team** | **¥6,000 × 5** | **¥720K** |
| **C. 中企業** | **30-100 人** | **採用しても定着しない、残業漬け** | **Max + seat** | **¥15,000 × 20** | **¥7.2M** |
| **D. 中堅** | **100 人〜** | **バックオフィス DX 推進中、予算あり** | **Enterprise** | **¥50,000 × 100** | **¥120M** |
| ラガード | 全規模 | IT を完全に拒否、紙と FAX 永遠 | (取らない) | — | — |

**1 件の D 層 = 4,000 件の A 層**。ユダヤ商法(金持ち相手に売る)的に、**A 層は撒き餌、B/C/D 層が本命**。

### 3.2 イノベーター理論レイヤー

本書第 3 章「イノベーター理論を集客に落とし込む」を FujiTrace に当てはめる:

| 層 | 検索行動 | FujiTrace での扱い |
|---|---|---|
| イノベーター(2.5%) | プロダクト名で検索 | 自作 / 自前 API。商工会議所に来ない。**少数だが拡散力**(X / Tech 系で拾う) |
| アーリーアダプター(13.5%) | 悩み・困りごとで検索 | 「AI 事務員」「AI 見積書」キーワードで流入。**SEO + LP デモ**で取る |
| **アーリーマジョリティ前期(0-10%)** | **逃げ場がなくなり相談に来る** | **商工会議所 / よろず支援 / Founder 営業**。本命ターゲット |
| アーリーマジョリティ後期 + レイトマジョリティ(58%) | 業界仲間が動くまで動かない | **紹介伝播**で後追い。Phase A2 以降 |
| ラガード(16%) | 永遠に動かない | **明示的に捨てる**(本書 第 3 章原則) |

84%(ラガード以外全層)を狙う本書の主張に従う。ただし**全層を同時には取りに行かない**。アーリーアダプター → アーリーマジョリティ → 後期マジョリティの**時系列の波**を作る。

### 3.3 撒き餌 → 本命の DRM 構造

本書第 6 章のフロントエンド / バックエンド理論に整合:

```
[フロントエンド = 撒き餌]
  Free 5h リセット        ← LINE Welcome Journey から流入
  Pro  ¥3,000/月          ← A 層(1人零細)、薄利、ボリューム勝負
                            目的は「次の人を紹介する仕組み」

[ミドル = 利益率の本体]
  Team ¥6,000/seat × 5    ← B 層(5-30人)、seat 課金の中核
                            「事務員を増員したいが採用無理」
  Max  ¥15,000 + seat     ← B/C 境界

[バックエンド = ユダヤ商法の "金持ち相手"]
  Enterprise ¥50,000〜    ← C/D 層(30人〜)
                            「採用予算は出るが定着しない」
                            LTV 億オーダー、M&A 価値の根幹
```

**「無料でもゴミは欲しがらない」** — Free / Pro はただの安売りではなく、**「明るい未来」を売る教材的フロントエンド**にする。LINE Welcome Journey / 4 章チュートリアル / 修了証 PNG はその実装。

### 3.4 取らない層(明示的に捨てる)

本書第 3 章「ラガードは切り捨てろ」原則:
- 紙と FAX 永遠の社長(全規模・16%)
- 「AI は信用ならん」と最初から拒否する層
- 製造業の工場系 IoT / 異常検知ニーズ(スコープ外、別事業)
- 人事健康管理 SaaS 領域(別事業)
- 士業独占業務領域(税務・登記・労務・契約書法的助言・診断・マイナンバー代行)

### 3.5 業種戦略(Phase 0 は絞らない、Phase A1 で逆引き)

Grok ペルソナ実験(2026-04-26)で AI が真っ先に選んだのは「**製造業 30 人**」。事前仮説としては製造業の適合度が高いが、商工会議所 / よろず支援拠点の現場では業種を選べないため、**Phase 0 中は業種を絞らずに営業・集客**する方針とする。

#### 事前仮説(参考、絞り込みではない)

| 業種 | 仮説適合度 | 理由 |
|---|---|---|
| 製造業 5-30 人 | ◎ | 人手不足深刻、紙 FAX Excel、後継者問題、ものづくり補助金、机上作業多 |
| 卸売・小売 5-30 人 | ○ | 在庫管理 + 請求書頻度高、AI 親和性高 |
| 建設・設備 5-30 人 | ○ | 見積書頻度高、現場と事務の二刀流社長多 |
| 飲食・サービス | △ | 売上規模小さい、IT 投資余力低い |
| IT・SaaS | × | 自前で AI 入れる、競合扱い |

#### Phase 0 の運用ルール(2026-04-27 確定)

- **業種を選ばずに営業 / LINE 流入 / オウンドメディアで取得を試みる**
- 取得した顧客の業種・規模・解約率・LTV を **逆引きで集計**
- Phase A1(2026-06-07〜) 時点でデータが揃っていれば、その業種に絞って業界別ランディング / 事例集 / 営業トークを最適化
- データが揃わなければ Phase A1 中も全業種で取得継続、Phase A2 以降に絞り込み判断

#### 業種を絞り込まない理由

1. 商工会議所 / よろず支援は来所者の業種を選べない
2. データが少ないうちに仮説で絞ると **アーリーアダプター起点を狭める**(イノベーター理論)
3. 1 件目の顧客の口コミ / 紹介は同業種に伝播するため、自然に業種が絞れていく
4. 仮説と実態がズレる可能性(製造業より建設業の方が刺さるかもしれない)を検証する余地を残す

つまり「製造業特化」は **Phase A1 以降の判断材料**として保留し、Phase 0 は実地検証フェーズに徹する。

## 4. 差別化(強烈な No.1 軸)

### 4.1 机上作業に特化(Founder 自己発見、本書第 5 章)

> **机上作業に特化、これには誰にも負けない、特化していて性能が悪いわけでもない**(Founder 2026-04-27)

汎用 AI(ChatGPT / Claude / Copilot)は**全体最適**を狙う。FujiTrace は**机上作業カテゴリの No.1** を狙う。Trout & Ries の Positioning 流カテゴリ作成。

### 4.2 オブザーバビリティ統合 = 安心の見えない基盤

詳細は **Section 6**。要点だけ:
- 顧客向け: 全実行が監査ログに残るから稟議が通る
- VC 向け: trace データが永続蓄積されるから M&A 価値が貯まる

### 4.3 日本ローカライズ 6 軸(Manus / グローバル汎用に対する非対称)

1. **日本語 PII 検出 15+ パターン**(氏名・住所・マイナンバー・在留カード等)
2. **日本商慣習対応**(押印・インボイス制度・支払サイト・源泉徴収・振込手数料負担)
3. **日本 SaaS 深連携**(freee / マネーフォワード / サイボウズ / Chatwork / LINE WORKS)

### 9.5 明るい未来の 3 主体ストーリー(営業 / LP 共通テンプレ)

#### 現状の中小企業(社長が共感する痛み)

**社長**:
- 朝 8 時に出社して見積書 / 請求書 / 経費 / メール返信で気づいたら昼
- 「経営判断する時間が欲しい」が口癖
- 子どもの保育園送迎にすら間に合わない
- 採用しても辞める、自分が雑務を抱える悪循環
- 求人広告に毎月 ¥10 万かけても応募ゼロ

**社員**:
- 単純作業の繰り返しで成長実感ゼロ
- 「自分の市場価値が下がっている」不安
- AI 時代に取り残される焦り
- でも何から始めていいか分からない

**事業**:
- 採用できない / 採れても辞める
- 受注はあるが処理しきれず断る
- 人手不足で残業漬け、離職スパイラル
- 業績は伸び悩み、後継者問題も近づく

#### 3 ヶ月後(おしごと AI フジ導入後)

**社長**:
- 雑務 8 割を AI に任せている
- 朝 8:30 に AI が「今日の生産計画」「リスク」「提案リスト」を提示
- 社長は 10 分で承認 → 残り時間は経営判断・営業・新規事業に全振り
- 早く家に帰れる、子どもと夕食を食べる
- 「AI 入れて本当に良かった」が口癖

**社員**:
- AI を「優秀な新入社員」として上司の立場で使いこなす
- 自分は判断 / 顧客対応 / 提案など、人間にしかできない仕事に専念
- AI 振り返りミーティング(毎週金曜 15 分)で AI のスキルを上げていく
- 「AI と共に働けるスキル」が身につき、市場価値が上昇
- 会社が成長していく実感がある

**事業**:
- 採用に依存しない事業設計が完成
- 離職率がほぼゼロに
- 人件費を増やさず処理量が 5-10 倍化
- 余った時間で新規事業 / 顧客深耕

#### 3 年後(長期未来)

**社長**:
- 売上 1.7 倍、営業利益率 8% → 15%
- 社員が「うちの会社、面白いことやってる」と胸を張る
- 経営判断のスピードと精度が劇的に向上
- 同業他社から「どうしてそんなに早く回るのか」と見られる存在に

**社員**:
- AI と共に働くスキルで市場価値が大企業の同職種を超える
- 業界内のロールモデルとして勉強会に呼ばれる
- 自分の会社に誇りを持つ

**事業**:
- 中堅企業を相手にしても「なんかこの会社違うな」と思わせる存在
- 「事務員月給 1/15 で AI 社員 15 人分働く」事例として業界誌に取材される
- M&A や上場の選択肢も視野に入る

#### 営業ピッチでの使い方

社長と話す時、現状の痛み(雑務 / 採用 / 残業)に共感を示してから、3 ヶ月後と 3 年後を**具体的な朝の風景**で描く。本書第 4 章「明るい未来は具体的な絵で」原則。

「社長、3 ヶ月後の朝 8:30 を想像してください。おしごと AI のダッシュボードが今日の優先事項を提示しています。社長は 10 分で承認、午前中はずっと経営判断と営業に時間を使えます。残業ゼロ、夜は子どもとご飯。**画面の隅にはカピぶちょーが「お疲れさんやで〜」と関西弁で励ましてくれる**。これを月 ¥3,000 ちょいから始められます。」

#### キーワード設計の原則

「悩みで検索する層を取る」が中核。ただし**単発ワードは競合が広すぎる**ため、**「悩み + 規模 + 解決手段ヒント」3 要素**で組み合わせる。

| 良い組み合わせ | 悪い組み合わせ |
|---|---|
| 人手不足 中小企業 解決 | 人手不足(競合無限) |
| 事務員 採用 できない | AI(競合無限) |
| 経理 自動化 中小企業 | 業務効率化(抽象) |
| 請求書 AI 中小企業 | DX(抽象) |
| 採用 諦めた 解決 | — |
| 社長 雑務 多すぎる | — |

#### リード磁石(本書第 6 章「無料でもゴミは欲しがらない」)

リード磁石は**「明るい未来」を約束する**もの。「資料 PDF」「無料相談」のような弱いものは作らない。

**キャラ完成前の暫定リード磁石**:

- 「人手不足の中小企業が読む、AI で月 ¥100 万浮かす実例集」(数字 + 具体性)

**キャラ完成後のリード磁石(本命)**:
- 「**おしごと AI フジが教える、人を雇わずに事務を回す方法**」(キャラ + 具体ベネフィット)
- 「**フジと一緒に始める、AI 社員 30 日チャレンジ**」(キャラ + 行動誘導)

#### 絶対ルール(Founder と約束済)

新規広告キャンペーン開始前に以下が**両方完了**していることが絶対条件(過去 3 キャンペーン ¥20K+ 出血の教訓):

2. **Keyword Planner 調査**: 検索ボリューム / CPC / 競合度を確認、競合無限のワードは避ける

両方が紙でレビューされていない状態で入稿するのは禁止。

### 戦略原則(普遍)

- 広告投下前はファネル設計 + Keyword Planner 必須(¥20K+ 出血の教訓)
- 責任閾値を超えない AI は商売にならない(要約 / 翻訳 / FAQ chatbot は着火剤にならない)
- 日本 BtoB SaaS は教育コンテンツで勝つ(HubSpot / freee 経営ハッカー / Salesforce Trailhead / AWS Training)
- Trace 多様性が Obs 商品価値の源泉(均質 trace = 商品力弱い)
- freee 非対称軸: freee は「記録・手続き・構造化・月額固定・現実主義」、FujiTrace は「判断・監査・非構造化・従量・理想主義」
- 「見せる」と「触らせる」を分離する(Benesse 年齢ゲート構造)
- **Design for Orchestration**: 全ツールは OpenAI function calling 互換スキーマで export(Phase A1 のエージェント実装時にゼロコストで繋がる)
- **二層メッセージング厳守**: 顧客 = シンプル訴求、VC = 構造的非対称
- **ウォンツの言葉で集客、ニーズの言葉で刺す**(本書第 4 章 + 2026-04-26 Founder 修正)
- **AI 社員を入口に Trace を取る、Trace で M&A 価値を作る**(バケツ穴修復後に初めて成立)
- **広告は直接 CV を狙うな、リスト集めに割り切れ**(過去 3 キャンペーン ¥20K+ 出血 + 2026-04-27 商工会議所アドバイス + 本書第 1/3/6/8 章が独立に同じ結論)
- **悩みで検索する層を「悩み + 規模 + 解決手段ヒント」3 要素のキーワード組合せで捕捉する**(単発ワードは競合無限)
- **AI とキャラを別レイヤーで並走させる**(AI = 標準語の実務担当、カピぶちょー = 関西弁の上司キャラ。1 キャラに統合すると業務文脈で関西弁混入の死亡パターンになる。現実のオフィスを画面内に再現する設計が正解)

## 16. VC / エンタープライズ向け追加ナラティブ

### 二層メッセージングの全体像

| 対象 | 使うナラティブ | 目的 |
|---|---|---|
| 顧客 / LP / 営業 | 「採用市場が壊れた日本で、AI 社員に机上作業を任せる」 | シンプル訴求、価格比較で勝つ |
| エンタープライズ / 監査担当 | 「AI 導入時の監査統制基盤が標準搭載」 | コンプラ予算から稟議通過 |
| **VC / 投資家** | **本 Section のフルナラティブ** | **差別化・モート・評価倍率向上** |
| 開発者 / OSS コミュニティ | 「LLM 観測の日本語標準 + Scaffolded Agent フレームワーク」 | 技術信頼性・エコシステム |

### VC 向けフルナラティブ

> **FujiTrace は、AI 社員を入口にして日本企業事務作業の trace データを独占蓄積する、データ / Obs プラットフォームである。**

#### 表層(顧客が見るもの)
- AI 社員(机上 AI)プラットフォーム
- 監査ログ標準搭載
- 5 プラン構造(Free / Pro / Team / Max / Enterprise)

#### 深層(VC が評価するもの)
1. **AI コスト提供 × Obs / データプラットフォーム収益化の二層構造**
2. freee に対する 5 軸非対称ポジション(判断 / 監査 / 非構造化 / 従量 / 理想主義)
3. 日本語 PII(15+ パターン)+ ドメスティックデータ + Scaffolded Agent の参入障壁
4. Claude Code / Operator / Manus 等のグローバル汎用エージェントが日本中小企業市場で勝てない構造的理由
5. **Trace 自動蓄積パイプラインの完全成立**(2026-04-26 バケツ穴修復、本日以降のデータが資産)

### データフライホイール

```
AI 社員が普及する
    ↓
Trace 大量蓄積(日本企業事務作業パターンの宝庫)
    ↓
 (a) LLM-as-Judge 精度向上 → プロダクト品質優位
 (b) 業界別ベンチマーク公開 → ブランド権威
 (c) B2B2B データ提供(保険 / 与信 / コンプラ) → 二次収益源
 (d) FujiTrace 専用モデル fine-tuning → 技術モート
    ↓
競合が時間でしか追いつけない構造的優位
```

### 評価倍率の二重構造

- **顧客獲得段階**: AI エージェント屋(10-30× ARR)で ARR を積み上げる
- **Exit 段階**: データ / Obs プラットフォーム(20-50× ARR)で評価される
- 同一 ARR でも Exit バリュエーションが大きく伸びる構造

### 日本 BtoB SaaS 勝ちパターンとの整合

- HubSpot(マーケティングツール販売 → 顧客データプラットフォーム化)
- Salesforce(SFA 販売 → CRM 統合プラットフォーム化)
- freee(会計ソフト販売 → 金融データプラットフォーム化)
- **FujiTrace = AI 社員販売 → 日本企業 AI 活用データプラットフォーム化**

### 禁止事項(二層メッセージング厳守)

- 顧客向け資料に「構造的非対称」「Obs プラットフォーム」「データモート」の語を出さない
- VC 向け資料に「¥3,000」「freee 代替」等の個別製品訴求を前面に出さない
- 価格は両者で共通だが文脈が違う:
  - 顧客 = 「会計ソフトと同じ価格で AI 事務員」
  - VC = 「ARR 積み上げアンカー」

## 19. 本書の運用ルール

### 19.1 Single Source of Truth

本書 `docs/戦略_2026.md` は **戦略レベルの単一情報源** とする。Founder + CEO AI + 各エージェントは本書を参照して動く。

### 19.5 2 キャラ並走モデルの口調混在禁止ルール(2026-04-27 確定)

**鉄則**: 「おしごと AI」と「カピぶちょー」の口調が混在するメッセージを生成しない。1 メッセージ 1 キャラの境界を厳密に維持する。

#### 違反パターン(NG)

| NG 例 | 理由 |
|---|---|
| AI が「お疲れさんやで〜、見積書を作りました!」 | AI は標準語の敬語のみ |
| AI が「了解しました、ええやんか〜」 | 関西弁混入 |
| カピぶちょーが「平素より大変お世話になっております」 | カピぶちょーは関西弁のみ |

#### 正しいパターン(OK)

```
[LINE メッセージ 1: AI レスポンス]
おしごと AI:
  見積書のチェックが完了しました。金額の整合性は問題ありませんが、
  支払条件の記載が抜けています。追加されることをお勧めします。

[LINE メッセージ 2: カピぶちょーフキダシ]
カピぶちょー(カピバラ):
  まいど〜、ええ感じや。支払条件さえ書いとけばバッチリやで〜☺
```

#### 実装責任

- SYSTEM_PROMPT 内で「あなたは AI です。関西弁を使ってはいけません」を明文化
- カピぶちょーは別 LLM 呼び出し or テンプレート関数で生成
- LINE では 2 メッセージ送信、Web ではマスコット領域でフキダシ表示
- 違反パターンが本番で検出されたら **即修正、緊急度高**

#### 例外: 業務文章への介入禁止

**カピぶちょーは絶対に業務文章を書かない**。ユーザーが「メールの文案作って」と頼んだ時、生成されるメール本文は標準語の敬語(AI 担当)。カピぶちょーは「ええ感じやで〜」とフキダシで一言添えるだけ。

```
[NG]
ユーザー: 「取引先に納期遅れのお詫びメール作って」
AI: 「平素より大変お世話になっております。さて、納期の件でお詫びさせて
     いただきますわ〜。ちゃうちゃう、もう少し丁寧に書きますね…」

[OK]
[AI レスポンス]
平素より大変お世話になっております。さて、納期の件でお詫び申し上げます。
誠に申し訳ございませんが、◯月◯日の納期につきまして、◯日ほど遅延する
見込みとなりました…

[カピぶちょーフキダシ]
ええメールやんか。お詫びはストレートに、原因と対策と新納期の 3 点
押さえとけば大丈夫やで〜
```
