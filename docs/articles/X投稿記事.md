## LLMアプリを本番運用して「地獄を見た」話と、そこから作ったもの

LLMを使ったアプリを本番に出すと、だいたい2週間くらいで地獄が始まる。

自分の場合はこうだった。

社内向けのRAGチャットボットを作った。開発時の精度テストは90%超え。自信を持ってデプロイした。

2週間後、こんな連絡が来た。

「なんか最近、回答の精度が落ちた気がするんですけど」
「先月のAPI利用料、想定の3倍なんですが」
「お客様の個人情報がプロンプトに入ってたらしいんですが、ログ残ってますか」

全部ヤバい。でも全部、従来のログ基盤では答えられない問いだった。

### なぜ従来の監視ツールでは無理なのか

普通のWebアプリなら、Datadogなりで500エラーを監視すれば大体なんとかなる。

LLMの厄介なところは、**ステータスコード200で正常に返ってきたレスポンスの中身が間違っている**こと。

```
ステータスコード: 200 OK
レスポンス: 「弊社の返品期限は購入後30日以内です」
実際の返品期限: 14日以内
```

これはエラーじゃない。障害でもない。でもユーザーからすれば完全に事故。

従来の監視は「システムが動いているか」を見る。LLMでは「出力が正しいか」まで踏み込む必要がある。ここが根本的に違う。

---

### LLM運用で本当に追うべきもの

自分が痛い目を見て学んだ、最低限追わないとまずい4つ。

**1. コスト**
LLMのAPI費用はユーザーの入力長に依存する。GPT-4とGPT-4o-miniで単価が20倍違う。月末に請求書を見て青ざめる前に、リアルタイムで追うしかない。

**2. ハルシネーション**
「正しそうに見えるけど間違っている」出力を、人間が全件チェックするのは不可能。LLMがLLMの出力を評価する「LLM-as-Judge」という手法で自動検知する。

**3. 日本語の個人情報**
マイナンバー、運転免許証番号、健康保険証番号。日本固有の個人情報パターンは海外のツールでは検知できない。LLMプロバイダーに送信される前に止める必要がある。

**4. エージェントの思考過程**
AIエージェントが「なぜその判断をしたか」を後から追えないと、問題が起きたときに原因を特定できない。

### で、作った

これらの課題を自分で解決するために、LLMオブザーバビリティプラットフォームを作った。

**FujiTrace** — 日本企業向けのLLM監視プラットフォーム。

何ができるかを簡単にまとめると：

- OpenAI / Claude / Gemini に対応。プロキシ型なので既存コードの変更なしで導入できる
- トークン消費・コストをリアルタイムで可視化。予算上限を設定してアラートを飛ばせる
- ハルシネーションをLLM-as-Judgeで自動検知
- 日本語の個人情報を15パターン以上自動検知して、LLMへの送信前にブロック
- AIエージェントの思考ステップをツリー表示で可視化

### なぜ国産にこだわったか

海外にはLangSmith、Langfuse、Braintrustなど、優れたプラットフォームがすでにある。この市場には600億円以上の投資が集まっている。

でも、日本企業が本番で使おうとすると壁がある。

- UIが英語のみ
- 日本語のPII検知に対応していない（マイナンバーなんて知らない）
- データが海外サーバーに保存される
- 日本語サポートがない

エンタープライズ企業にとって、この4つは「あったらいいな」ではなく「ないと導入できない」レベルの要件になる。

だから作った。
