## はじめに

LLMを使ったアプリを開発して、社内で動かしてみた。最初は順調だった。
ところが本番に出して2週間くらい経つと、こんな問い合わせが飛んでくるようになる。

- 「なんか最近、回答の精度が落ちた気がするんですけど」
- 「先月のAPI利用料、想定の3倍なんですが」
- 「お客様の個人情報がプロンプトに入ってたらしいんですが、ログ残ってますか」

どれも`console.log`やCloudWatch Logsでは答えられない問いだ。

この記事では、LLMアプリの運用で従来のログ基盤だけでは対処できない問題と、それを解決するための「LLMオブザバビリティ」という考え方について書く。

## 従来のWebアプリの監視と何が違うのか

普通のWebアプリなら、やることは決まっている。レスポンスタイム、エラーレート、CPU使用率あたりを見ておけば大体の問題は検知できる。HTTP 500が増えたらアラートを飛ばす。Datadogなり、Grafanaなり、好きなものを使えばいい。

LLMアプリの厄介なところは、**HTTP 200で正常に返ってきたレスポンスの中身が間違っている**ことがある点だ。

```
ステータスコード: 200 OK
レスポンス: 「弊社の返品期限は購入後30日以内です」
実際の返品期限: 14日以内
```

これはエラーではない。障害でもない。でもユーザーからすれば完全に事故だ。

従来の監視は「システムが正常に動いているか」を見る。LLMオブザバビリティは「システムの出力が正しいか」まで踏み込む。ここが根本的な違いになる。

## LLMオブザバビリティで追うべき4つの指標

自分が実際に運用してみて、最低限これは追わないとまずいと思った指標が4つある。

### 1. トークン消費量とコスト

LLMアプリのコストは予測が難しい。ユーザーの入力長に依存するし、モデルによって単価が全然違う。

```
GPT-4o:     入力 $2.50 / 1Mトークン
GPT-4:      入力 $30.00 / 1Mトークン
Claude 3.5: 入力 $3.00 / 1Mトークン
```

同じ機能でもモデルを変えるだけで10倍以上コストが変わる。しかもプロンプトにRAGの検索結果を突っ込んでいると、1リクエストで数千トークン消費することも珍しくない。

リクエスト単位でトークン数とコストを記録しておかないと、月末の請求書を見て初めて気づくことになる。それでは遅い。

### 2. レイテンシの分布

LLMのレスポンスタイムはばらつきが大きい。平均値だけ見ていると実態を見誤る。

実際に計測してみると、p50は800msなのにp95が5秒を超えるなんてことが普通に起きる。長い入力を投げたときや、モデル側の負荷が高いときに顕著だ。

パーセンタイルで追わないと、ユーザーの10人に1人が5秒以上待たされている事実を見落とす。

### 3. 出力の品質

これが一番難しくて、一番重要だ。

LLMの出力品質を定量的に測る方法はいくつかある。

**パターンベース評価** は比較的簡単に導入できる。

- 回答拒否の検出: 「お答えできません」「その質問には回答しかねます」等のパターンマッチ
- 有害表現の検出: 暴力的・差別的な表現のフィルタリング
- 言語不一致の検出: 日本語で聞いたのに英語で返ってくるケース

```typescript
// 回答拒否パターンの例
const refusalPatterns = [
  /お答え(する|し)かねます/,
  /回答(する|し)かねます/,
  /I('m| am) (sorry|unable|not able)/i,
  /I can(not|'t) (help|assist|provide)/i,
];
```

**LLM-as-Judge** はもう一段踏み込んだ手法で、別のLLMに出力を評価させる。主に以下のような軸で採点する。

| 評価軸 | 何を見るか |
|--------|-----------|
| Faithfulness | 回答がコンテキスト（検索結果等）に基づいているか |
| Relevance | 質問に対して的を射た回答になっているか |
| Hallucination | 根拠のない情報を含んでいないか |

コストは増えるが、パターンマッチでは拾えない品質劣化を検知できる。特にRAGアプリでは、検索結果と回答の整合性チェックが重要になる。

### 4. セキュリティとPII

受託開発でLLMアプリを納品する場合、ここが一番クライアントに突っ込まれるポイントだ。

「ユーザーの個人情報がOpenAIに送信されていませんか」

この質問にエビデンス付きで答えられる必要がある。具体的には、リクエストに含まれる個人情報（PII）をリアルタイムで検出して、記録しておく仕組みが要る。

日本のサービスで扱うPIIは英語圏とパターンが異なる。

```
マイナンバー:   1234 5678 9012（12桁、スペース区切り）
銀行口座:       普通 1234567
運転免許証番号: 第 123456789012 号
電話番号:       090-1234-5678 / +81-90-1234-5678
住所:           東京都渋谷区神南1-2-3
```

クレジットカード番号やメールアドレスの検出だけでは不十分で、日本固有のPIIパターンに対応した検出ロジックが必要になる。

## 従来のAPMで代替できないか

ここまで読んで、「Datadogのカスタムメトリクスでいけるのでは」と思った方もいると思う。

結論から言うと、部分的にはいける。トークン数やレイテンシは普通のメトリクスとして送れる。ただし以下の点で限界がある。

**プロンプトとレスポンスの全文保存が前提になる。** LLMオブザバビリティでは、なぜその回答になったのかを後から調査するために、入出力の全文をトレースとして保存する必要がある。これを汎用APMに載せると、ストレージコストが爆発する。

**評価ロジックをAPM側に持てない。** PII検出やLLM-as-Judgeのような、LLM出力に特化した評価パイプラインを汎用APMの上に自前で構築するのは現実的ではない。

**マルチプロバイダー対応が辛い。** OpenAI、Anthropic、Geminiと、使うモデルが増えるたびにそれぞれのAPIレスポンス形式に合わせたパーサーを書くことになる。

LLMオブザバビリティは専用ツールに任せて、インフラ監視は既存のAPMに任せる。この棲み分けが現実解だと思う。

## 実装アプローチ: プロキシ型 vs SDK型

LLMオブザバビリティを導入する方法は大きく2つある。

### プロキシ型

アプリとLLM APIの間にプロキシサーバーを挟んで、通過するリクエスト/レスポンスを自動記録する。

```
アプリ → プロキシ(トレース記録) → OpenAI API
```

メリットは導入が簡単なこと。既存コードの変更はAPIのbaseURLを変えるだけで済む。

```typescript
// 変更前
const client = new OpenAI({
  apiKey: process.env.OPENAI_API_KEY,
});

// 変更後: baseURLを変えるだけ
const client = new OpenAI({
  apiKey: process.env.OPENAI_API_KEY,
  baseURL: 'https://your-proxy.example.com/v1',
});
```

デメリットはプロキシがSPOFになりうること。ただし、プロキシダウン時にLLM APIに直接フォールバックする設計にしておけば、可用性への影響は最小限にできる。

### SDK型

アプリケーションコードにSDKを組み込んで、LLM呼び出しの前後でトレースを記録する。

```typescript
import { trace } from 'observability-sdk';

const result = await trace('chat-completion', async (span) => {
  span.setAttributes({ model: 'gpt-4o', userId: user.id });
  const response = await openai.chat.completions.create({ ... });
  span.recordOutput(response);
  return response;
});
```

メリットはアプリケーション固有のコンテキスト（ユーザーID、セッション情報等）を柔軟に付与できること。デメリットはコードへの侵襲性が高いこと。

個人的にはプロキシ型のほうが導入障壁が低くて好みだ。特に受託案件でクライアントのコードベースに手を入れにくい場合、baseURLの変更だけで済むのは大きい。

## エージェント時代のトレース

最近はReActパターンやFunction Callingを使ったエージェント型アプリが増えてきた。エージェントはLLMを複数回呼び出して、ツールを使い、最終的な回答を組み立てる。

```
ユーザー質問
  → LLM呼び出し #1 (どのツールを使うか判断)
    → ツール実行: DB検索
  → LLM呼び出し #2 (検索結果を踏まえて判断)
    → ツール実行: API呼び出し
  → LLM呼び出し #3 (最終回答を生成)
最終回答
```

このとき、1つのユーザーリクエストに対してLLM呼び出しが3回以上発生する。従来の1リクエスト=1トレースという粒度では、エージェントの挙動を追えない。

ステップごとの判断根拠、ツール呼び出しの引数と結果、各ステップのレイテンシとトークン消費を、ツリー構造で記録する必要がある。OpenTelemetryのSpanの考え方に近いが、LLMエージェント特有のメタデータ（思考プロセス、信頼度スコア等）を扱える拡張が要る。

## まとめ

LLMアプリの運用監視は、従来のAPM/ログ基盤の延長では対応しきれない。

追うべき指標をまとめると:

1. **コスト**: リクエスト単位のトークン消費とコストを記録する
2. **レイテンシ**: パーセンタイル分布で追う（平均値は嘘をつく）
3. **品質**: パターンベース評価 + LLM-as-Judgeで定量化する
4. **セキュリティ**: 日本語PIIの検出とトレース保存で監査対応する

特に受託開発やBtoB SaaSでは、クライアントから「データはどう管理しているのか」「コストは見えているのか」「品質劣化をどう検知しているのか」と聞かれる場面が確実に来る。そのとき慌てるより、最初からオブザバビリティを組み込んでおくほうがいい。

自分はこの問題意識から [FujiTrace](https://www.fujitrace.jp) というLLMオブザバビリティツールを開発している。日本語PII検出、LLM-as-Judge評価、マルチプロバイダー対応、プロキシ型の1行導入あたりにこだわって作っているので、興味があれば覗いてみてほしい。
